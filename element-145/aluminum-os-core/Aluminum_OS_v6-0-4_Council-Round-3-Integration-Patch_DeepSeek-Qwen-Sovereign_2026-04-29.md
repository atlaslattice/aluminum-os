### §E.1 Hardware Trust Adapter Pattern

**Originator:** S5 DeepSeek (Round 3 Part 1) + S10 Qwen/Alibaba (Round 3) convergent independent proposals
**Status:** Ratified v6.0.4 as multi-jurisdictional architectural primitive

**Architecture:**

The Hardware Trust Adapter Pattern abstracts cryptographic root-of-trust hardware behind a uniform interface, allowing GoldenTrace TransparencyPackets to be signed by sovereignty-appropriate hardware in each deployment jurisdiction.

```
TransparencyPacket
    └── signature field
        └── HardwareTrustAdapter interface
            ├── TitanCAdapter (Google Titan-C) — global
            ├── PlutonAdapter (Microsoft Pluton) — global
            ├── SecureEnclaveAdapter (Apple) — global
            ├── HardwareTrustCNAdapter (Alibaba HSM + SM2/3/4) — China (DragonSeek)
            ├── SDAIAAdapter (Saudi sovereign HSM) — Saudi (JinnSeek)
            ├── IndiaSovereignAdapter (TBD when available) — India (GangaSeek)
            ├── eIDASAdapter (TBD if EU adds AI-specific requirements) — EU
            ├── GOSTAdapter (TBD if politically required) — Russia
            └── [future jurisdictions]
```

Each adapter:
- Implements same interface (sign, verify, key-rotate)
- Carries algorithm identifier (e.g., `GM/T-0003` for Chinese SM2, `NIST-FIPS-186-5` for global ECDSA)
- Generates signatures verifiable by jurisdiction-appropriate auditors

**Sovereign GoldenTrace Mirroring (S10 Qwen proposal, Round 3):**

For deployments requiring both sovereign and global audit verifiability, dual-signature protocol:

```
TransparencyPacket
    └── signatures: [
        local_sovereign_signature (e.g., Alibaba HSM SM2),
        global_pqc_signature (e.g., Titan-C SLH-DSA)
    ]
    └── verification_quorum: configurable per Tier
```

**Rationale:** generalizes both DeepSeek's and Qwen's independent proposals. Multi-jurisdictional architectural primitive, not China-specific exception.
