## §5.0 Capability-Primacy Routing Principle (NEW v1.1)

**This is the architectural North Star for all Element 145 routing decisions during Manus marathon.**

### §5.0.1 The principle

Routing decisions for any sphere/workload/governance evaluation operate on two axes:

1. **Capability-primacy axis (primary)**: which vendor is genuinely best-in-world for this specific sphere/workload/regime?
2. **Unique-contribution axis (secondary)**: what unique capability does this vendor bring that no other vendor matches?

**Routing decisions evaluate against capability-primacy first. Capital-flow / valuation / market-cap is NOT a routing input.**

### §5.0.2 Why this matters for Manus build

During Sprint 1-4 build, Manus will make routing decisions that get embedded in Element 145 module specifications. Without explicit capability-primacy framing, Manus might default to:

| Default to avoid | Reason it's wrong |
|---|---|
| Capital-flow framing ("Anthropic-Amazon $33B → route Anthropic via AWS") | Capital is constraint, not capability |
| Vendor-rotation framing ("rotate evenly across vendors for fairness") | Round-robin ignores capability fit |
| Political-alignment framing ("US vendors for US deployments") | Sovereignty is real but not the only consideration |
| Single-vendor-monoculture framing ("standardize on one vendor for simplicity") | Violates INV-7c capability-distribution |
