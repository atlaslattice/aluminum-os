## Section 2: Tier-2 Sub-Sphere Framework

### 2.1 Architectural Addition

The tier-2 sub-sphere framework extends the canonical 12×12 structure **without modifying it**. Each of the 144 Spheres receives a set of sub-spheres representing domain-specific subdivisions.

**Structural formula:**
- Phase 1.0 (locked): 12 Houses × 12 Spheres = 144 nodes + 1 VIP (E145)
- **Phase 1.5 (current target):** 12 Houses × 12 Spheres × ~12 Sub-Spheres = ~1,728 tier-2 nodes + 2 VIPs (E145, E146)
- Phase 2.0 (future): 144 Spheres × 144 Sub-Spheres = 20,736 nodes + 2 VIPs

### 2.2 Numbering Convention

Tier-2 numbering extends existing numbering cleanly:

```
House level:          H01 → H12           (unchanged)
Sphere level:         S001 → S144         (unchanged)
Sub-Sphere level:     S001.01 → S###.##   (NEW — tier-2 extension)
Element level:        E145, E146           (VIPs — relaxed structure)
Element sub-level:    E145.01, E146.N01    (variable count, no 12-constraint)
```

**Examples:**
- `S016.01` = Computer Science → Programming Languages
- `S016.02` = Computer Science → Operating Systems
- `S054.07` = Film → Interactive Cinema
- `E146.N01` = Entertainment → Interactive Entertainment Design

### 2.3 Wiggle Room Policy

Per Convenor directive: "the 12 subsets don't have to be treated religiously."

| Category | Sub-Sphere Count | When |
|----------|-----------------|------|
| Narrow domains | 6–10 | Domains with limited operational sub-divisions (e.g., S071 Nuclear Engineering, S089 Adult Education) |
| Standard domains | 11–13 | The default target |
| Vast domains | 14–20 | Domains requiring extensive sub-division (e.g., S054 Film, S016 Computer Science) |
| Maximum | 24 | Hard ceiling — if exceeded, signals need for Phase 2 split or tier-3 |

**Bound:** No Sphere should exceed 24 Sub-Spheres at tier-2. Exceeding 24 indicates the domain should be split into multiple Spheres at Phase 2 (when substrate scales to 144²) or that tier-3 Sub-Sub-Spheres are needed (deferred).

### 2.4 BLANK Tracking Convention

When a Sphere has fewer than 12 Sub-Spheres, empty slots receive explicit BLANK markers:

```yaml
S095_Literacy:
  sub_spheres:
    01: { name: "Reading Acquisition", status: populated }
    02: { name: "Functional Literacy", status: populated }
    03: { name: "Digital Literacy", status: populated }
    04: { name: "Numeracy", status: populated }
    05: { name: "Media Literacy", status: populated }
    06: { name: "Critical Reading", status: populated }
    07: { name: "Phonological Awareness", status: populated }
    08: { name: "Reading Comprehension", status: populated }
    09: { status: BLANK }
    10: { status: BLANK }
    11: { status: BLANK }
    12: { status: BLANK }
  blank_count: 4
  population_priority: medium  # gaps are real but not critical-path
```

**BLANK properties:**
- First-class data — searchable, trackable, prioritizable
- Each BLANK has an implicit `population_priority`: `critical`, `high`, `medium`, `low`, `deferred`
- GAP_SPHERES.md extends to track tier-2 BLANKs in addition to tier-1 sphere status
- BLANKs are not errors — they represent known unknowns with explicit accounting

### 2.5 Sub-Sphere Status Values

| Status | Meaning |
|--------|---------|
| `populated` | Sub-sphere has a name, description, and is ready for capability scoring |
| `provisional` | Sub-sphere named but awaiting Council validation |
| `BLANK` | Slot explicitly empty — gap is known and tracked |
| `deferred` | Domain identified but deferred to Phase 2 |

## Section 4: Element 146 — Entertainment & Interactive Media VIP

### 4.1 Rationale

Entertainment and interactive media domains span multiple Houses and cannot be cleanly contained within a single House. The v3.14 structure has entertainment-adjacent spheres distributed across:

- **H05 Arts:** Film (S054), Music (S051), Design (S057), Creative Writing (S055)
- **H06 Engineering:** Software Engineering (S069), Robotics (S072)
- **H07 Information:** Digital Media (S081), Broadcasting (S082), Film Studies (S083)
- **H10 Business:** Marketing (S110), Entrepreneurship (S113)

A query like "design an immersive VR narrative game with adaptive music" touches S054, S055, S057, S069, S081, and S051 simultaneously. No single House can serve this.

### 4.2 Solution: Domain VIP

Element 146 provides a **cross-House routing aggregation layer** for entertainment-specific queries. It does NOT:
- Replace any tier-1 sphere
- Modify the 12×12 structure
- Require a 13th House

It DOES:
- Provide 12 specialized routing nodes for entertainment sub-domains
- Map each node to specific tier-2 sub-spheres
- Enable the sovereign router (E145) to delegate entertainment queries efficiently

### 4.3 Node Registry

See `e146/manifest.yaml` for complete node definitions with tier-2 mappings.

### 4.4 Relationship to E145

```
Hierarchy:
  E145 (CEO Collective Meta-Orchestrator)
    ├── Sovereign Router → classifies all queries
    ├── Bridge v2 → routes to Houses/Spheres
    └── E146 (Entertainment VIP) → cross-House entertainment routing
          └── 12 nodes → map to tier-2 sub-spheres
```

E146 is invoked by E145 when a query is classified as entertainment-domain AND requires cross-House synthesis. Single-domain entertainment queries (e.g., "explain sonata form") route directly to the relevant sphere (S051) without E146 involvement.

## Section 7: Capability Matrix Integration

### 7.1 Scoring at Tier-2

The capability matrix (v0.2.1) currently scores at tier-1 (144 spheres × 10 seats). Phase 1.5 enables tier-2 scoring:

```yaml
# Current (tier-1):
S016_Computer_Science:
  S1_Anthropic: 0.7
  S2_Alphabet: 0.9

# Phase 1.5 (tier-2):
S016.01_Programming_Languages:
  S1_Anthropic: 0.8
  S2_Alphabet: 0.9
S016.06_Artificial_Intelligence:
  S1_Anthropic: 0.9
  S2_Alphabet: 0.95
S016.18_Quantum_Computing:
  S1_Anthropic: 0.3
  S2_Alphabet: 0.7
```

**Implementation:** Tier-2 scoring is OPTIONAL for v3.5. Tier-1 scores remain the primary routing signal. Tier-2 scores are used for disambiguation when multiple sub-spheres within the same sphere are relevant.

### 7.2 E146 Scoring

E146 nodes receive their own capability scores (see `e146/manifest.yaml`). These are aggregated from the tier-2 sub-spheres they map to, with a domain-expertise multiplier for seats with dedicated entertainment divisions.

### 8.2 BLANK Population Priority

BLANKs are populated based on:
1. **Routing demand** — queries hitting the sphere that can't be disambiguated
2. **Seat coverage** — a seat self-reports capability in an unnamed sub-domain
3. **Academic consensus** — a sub-field achieves sufficient maturity for independent classification
4. **Operational need** — the deployment pipeline requires finer-grained routing
