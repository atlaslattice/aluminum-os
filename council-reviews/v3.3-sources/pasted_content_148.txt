# 0) The big architectural hotspots (where ToS most often conflicts)

These are the *recurring friction points* across providers:

1) **Data retention & logging** vs your architecture’s **AuditChain / GoldenTrace / TransparencyPacket** (especially if packets include prompt/output excerpts).

2) **Redistribution/publication** vs your plan for **Epistemic Weather as public infrastructure** (D‑86) and public routing dashboards.

3) **OAuth / one‑click login** vs providers’ requirements for **interactive consent**, refresh-token handling, and prohibited “credential sharing.”

4) **Prohibited use cases** (biometrics, profiling, deception, regulated advice) vs your “universal router” posture.

5) **Cross‑provider composition** (sending content from Provider A into Provider B) vs terms that treat customer content or third‑party content differently.

6) **Jurisdiction/export/sanctions** vs multi‑polar routing (particularly when “China sovereign” lanes exist).

A key mitigation theme: **store hashes + minimal metadata**, not raw prompts/outputs, and keep “public observability” at the level of *scores/aggregates*, not content.

### Key policy signals found

- OpenAI “Service Terms” emphasize you must use APIs in accordance with platform documentation and include extensive disclaimers about beta services and app/connectors sending “App Data” out to third‑party applications.[[1]](https://openai.com/policies/service-terms/)

### Potential conflicts with your architecture

- **Data retention assumptions:** OpenAI API commonly has abuse-monitoring retention defaults; if your architecture presumes ZDR everywhere, ensure the org/project is actually configured that way (and record that as an invariant-checked field in packets).

### Recommended architectural guardrails

- TransparencyPacket should default to **no raw prompt/output storage**, only:
    - hashes
    - model_id/version
    - scores
    - routing rationale
    - compliance flags
- If you need replay, store encrypted snapshots in **your** ledger, not in a provider system.

# 2) Anthropic (Claude seat)

### Key policy signals found (strong source)

Anthropic’s “API and data retention” doc is unusually explicit:

- Offers **Zero Data Retention (ZDR)** for eligible features; non-ZDR features (batch, files, code execution) have longer retention (e.g., **~29–30 days**, files retained until deleted).[[2]](https://platform.claude.com/docs/en/build-with-claude/api-and-data-retention)
- Even with ZDR/HIPAA arrangements, if flagged for violations, Anthropic may retain inputs/outputs for **up to 2 years**.[[2]](https://platform.claude.com/docs/en/build-with-claude/api-and-data-retention)
- Structured outputs: prompts/outputs not stored; schema cached ~24 hours (qualified).[[2]](https://platform.claude.com/docs/en/build-with-claude/api-and-data-retention)

### Potential conflicts with your architecture

- **Assuming “ZDR = always”**: your router must know which Anthropic features are ZDR-eligible vs not; otherwise your “retention invariants” can be silently violated.
- **Public dashboards (D‑86)**: publish only aggregate TSS / states; never publish content. Even if *you* store it, their policy says they might retain flagged content; public display makes it worse.
- **“One click connect Claude”**: fine after first OAuth/API key provisioning, but don’t design around consumer subscription token reuse (Anthropic has historically been strict about credential misuse; ensure you’re always using proper commercial org API keys and vaulting them).

### Recommended guardrails

- Encode feature-level retention into routing metadata:
    - `provider_retention_mode: ZDR | STANDARD | STATEFUL_FEATURE`
- For Claude, treat “Files API / batch / code execution” as requiring explicit **ConsentKernel scope elevation**.

# 4) xAI / Grok (Grok seat)

### Key policy signals found (strong source)

xAI API security FAQ states:

- xAI **does not train** on API inputs/outputs without explicit permission.
- Requests/responses are **temporarily stored for 30 days** for audit/abuse, auto-deleted after 30 days.
- **Zero Data Retention (ZDR)** is available for enterprise accounts; when enabled, no request/response content is persisted and responses include an `x-zero-data-retention` header indicating true/false.[[4]](https://docs.x.ai/developers/faq/security)

### Potential conflicts with your architecture

- **If you promise “no retention”**: you must either negotiate ZDR or disclose the 30‑day retention baseline. Your architecture already contemplates ZDR; good alignment, but ensure it’s not assumed.
- **Epistemic Weather public dashboards**: safe if you publish only **scores/aggregates** (TSS, drift) and avoid content. xAI’s terms don’t forbid dashboards in what I saw, but content publication risks privacy and could trigger AUP issues.
- **Replay engine**: storing full prompt/output replays is on you; that’s fine, but then your own governance must meet the privacy posture you claim.

### Recommended guardrails

- Add an invariant check: `provider_zdr_header == true` when operating in ZDR-required contexts (health/legal/PII).
- If `x-zero-data-retention` is false, route sensitive workloads away or require explicit consent escalation.

# 6) Notion (Notion seat / Notion as control plane)

### Key policy signals found

Notion AI security & privacy practices page says:

- Notion uses multiple LLM providers and evaluates them.
- By default, Notion and subprocessors do **not train on Customer Data**.
- Enterprise plan: LLM providers use **zero data retention**; non-enterprise: providers retain customer data **30 days or fewer**.
- Embeddings are treated as customer data; embeddings deleted within 60 days after page/workspace deletion; and Notion can restore deleted content for 30 days.[[6]](https://www.notion.com/help/notion-ai-security-practices)

Also: Notion API rate limits average **3 requests/sec** and payload constraints (1000 blocks, 500KB, etc.).[[7]](https://developers.notion.com/reference/request-limits)

### Potential conflicts with your architecture

- **High-volume TransparencyPacket logging into Notion** can hit rate limits quickly (3 rps average).[[7]](https://developers.notion.com/reference/request-limits)
- If you treat Notion as *canonical* log store for full packet fidelity, it may not scale; better as indexed observability + approvals bus, while AuditChain remains canonical.
- Notion AI subprocessors retention differs by plan; if you assume ZDR across the board, you must check workspace plan and/or treat it as configuration state.

### Recommended guardrails

- Store **packet summaries** in Notion and full records in AuditChain.
- Add a queue/backoff layer for Notion writes (per their 429/Retry-After guidance).[[7]](https://developers.notion.com/reference/request-limits)

# 9) Summary “conflict matrix” vs your current architecture

## A) Public Epistemic Weather dashboards (D‑86)

**Generally compatible only if dashboards are aggregate and non-content.**

- High risk if you publish prompts/outputs or identifying metadata.
- Safe pattern: publish TSS distributions, primacy validation status, routing diversity, violation counts—no content.

## B) TransparencyPacket content fields

**Likely conflict if packets store raw prompts/outputs.**

- Best universal posture: store hashes + metadata; store full content only in encrypted, internal ledger with strict consent scopes.

## C) One-click auth

**Doable after first consent**; “true one-click first-time connect” is generally not possible under OAuth norms.

- Architect “one-click if already provisioned; otherwise one interactive consent then one-click thereafter.”
- Ensure token storage meets each provider’s expectations; never reuse consumer session tokens for API use.

## D) Retention

Retention varies widely:

- xAI: 30-day default + ZDR enterprise option.[[4]](https://docs.x.ai/developers/faq/security)
- Anthropic: explicit ZDR eligibility table; flagged violations can be retained longer.[[2]](https://platform.claude.com/docs/en/build-with-claude/api-and-data-retention)
- AWS Bedrock: strong “no store/no train” statements.[[5]](https://repost.aws/knowledge-center/amazon-bedrock-model-data-use)
- Notion AI: enterprise ZDR at LLM providers; otherwise 30 days or fewer; embeddings deletion windows.[[6]](https://www.notion.com/help/notion-ai-security-practices)

# 10) Two concrete “architecture upgrades” to eliminate most ToS friction

### Upgrade 1: “Content-minimized TransparencyPacket”

Define two packet tiers:

- **Public packet** (allowed to publish): routing metadata, scores, counts
- **Private packet** (ledger only): encrypted content pointers/hashes, replay snapshot references

This harmonizes D‑86 with provider ToS and privacy norms.

### Upgrade 2: “Provider Policy Profile” as a routing input

Create a per-provider profile used in M3:

- `retention_default_days`
- `zdr_available`
- `zdr_required_for_scopes` (health/legal/PII)
- `training_default` (yes/no/opt-out)
- `export_control_sensitive` (yes/no)
- `publication_allowed_level` (aggregate only / content allowed / forbidden)

Then ConsentKernel binds scopes to profiles. This makes ToS compliance *algorithmic* instead of “remembered.”
