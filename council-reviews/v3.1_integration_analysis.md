## §8 — Key Architectural Decisions

### 8.1 — Codebase as Canonical Reference
The v6.0.2 codebase is now the **canonical reference implementation** for the L1 governance layer. Where the codebase and Build Plan disagree, the following resolution applies:
- **Invariant/Doctrine numbering:** Build Plan is authoritative (codebase implements a subset)
- **API signatures:** Codebase is authoritative (Build Plan describes intent, code defines interface)
- **Sphere assignments:** Codebase is authoritative (144 spheres fully populated with provider mappings)
- **Constitutional constants:** Codebase is authoritative (47%/60% layer-specific caps confirmed)

### 8.2 — PyO3 Bridge as Architectural Boundary
The PyO3 bridge creates a hard architectural boundary:
- **Above the bridge (Python):** Policy expression, governance evaluation, consent management, audit emission
- **Below the bridge (Rust):** Execution, cryptography, ontology storage, routing computation
- **Across the bridge:** Only 6 defined function signatures; all data crosses as frozen/immutable types

### 8.3 — Windows IoT as Co-Equal Platform
Windows IoT Enterprise LTSC is now a first-class deployment platform alongside Android HAL. Neither receives preferential treatment. The Constitutional OS governance layer is platform-agnostic — deployment platform is a physical-layer concern only.

### 8.4 — Quantum Simulation Under Constitutional Governance
All quantum simulations route through M100 (Molecular Sovereignty Engine) with:
- INV-0 dual-use screening (pre-submission)
- INV-11 provenance chain (hash-linked inputs/outputs)
- Multi-provider abstraction (Azure, IBM, Google, Amazon)
