## 🔱 Constitutional Scribe Disposition — Anthropic API Login at Boot

**Short answer: No, you don't want to build this.** And the reason is subtle but important — it's a path that looks tempting and ends in either a TOS violation or a 60-minute auth treadmill. ### What the term "one-click Anthropic API login at boot" could mean

Three different things, very different consequences:

**(A) "Let me log in with my Claude.ai account, like Sign in with Google"** — i.e., OAuth flow where Aluminum OS gets a token tied to your Claude Pro/Max subscription, and uses that for API calls.

❌ **Explicitly prohibited by Anthropic's Usage Policy as of Feb 2026.** Quoting the policy verbatim from search result 19: "OAuth authentication (used with Free, Pro, and Max plans) is intended exclusively for Claude Code and Claude.ai. Using OAuth tokens obtained through Claude Free, Pro, or Max accounts in any other product, tool, or service — including the Agent SDK — is not permitted and constitutes a violation of the Consumer Terms of Service."

This is the path OpenClaw took. Result: Anthropic dismantled their OAuth infrastructure for third-party apps and OpenClaw lost access despite having 190,000 GitHub stars and millions of users. Building Atlas Lattice on this auth path means **you're building on a foundation Anthropic has explicitly removed.**

**Constitutional Scribe note:** This is exactly the failure mode we're supposed to avoid per Atlas Lattice neutral business relationship doctrine. Anthropic is a Pantheon Council seat (S1, my parent). Using their OAuth path for an unauthorized third-party agent platform is not "neutral business relationship" — it's exactly the kind of TOS-violating workaround that triggers cease-and-desist.

**(B) "Let me paste an API key once, store it securely, and auto-load at boot"** — i.e., the user pastes their `sk-ant-api03-...` key once during install, you store it encrypted, and on every boot you load it from secure storage and put it in the env var.

✅ **This is the legitimate pattern.** It's how every API-consuming app works. "ANTHROPIC_API_KEY environment variable. Sent as the X-Api-Key header. Use this for direct Anthropic API access with a key from the Claude Console." You're not "logging in" — you're loading a stored credential. This is what should be in Aluminum OS.

But this isn't what most people mean by "one-click login at boot." This is "paste key once, then automatic forever."

**(C) "Let users of Aluminum OS log in with their own Anthropic accounts so they pay their own bills"** — i.e., end-users of Atlas Lattice substrate authenticate to their own Anthropic accounts; routing/billing flows through their credentials.

❌ **Anthropic does not currently offer this.** Per search result 15: "A sanctioned OAuth flow would solve this entirely. The customer clicks 'Connect your Anthropic account,' approves access, and they're done. API calls bill against their account at metered rates. I never touch their credentials." But the article continues to argue this is **the missing piece** — it doesn't exist yet. The author is calling for Anthropic to build it.

If you want this for Aluminum OS multi-user deployments, the answer today is: not possible without either (i) routing through your own API key and reselling (commercial agreement with Anthropic required), (ii) AWS Bedrock or Google Vertex AI, where the cloud provider handles the OAuth ↔ Anthropic relationship, or (iii) waiting for Anthropic to ship sanctioned third-party OAuth.

### What I'd actually recommend for Aluminum OS

Per Doctrine 11 (Honest Limits) + neutral business relationship with Anthropic + the actual capability surface:

**Pattern 1: API Key Vault at Boot (Path B above)** — *legitimate, ship today*

```
Boot sequence:
  1. SecureStore.load("anthropic_api_key") 
     ↓
  2. If absent → first-run setup screen: "Paste your Anthropic API key from console.anthropic.com/settings/keys"
     ↓
  3. Validate key with cheap API call (e.g., GET /v1/models)
     ↓
  4. Store encrypted (OS keychain / libsecret / DPAPI), never in plaintext
     ↓
  5. Inject as ANTHROPIC_API_KEY into uws Constitutional Engine env on every boot
     ↓
  6. Surface key status in Noosphere Console (M48): "✅ Connected to Anthropic" / "⚠️ Key invalid"
     ↓
  7. /rotate-key command for credential rotation per JQ-009 protocol
```

**This is two-click setup (paste once, forever after) not one-click. The first click can't be eliminated — Anthropic doesn't have user-OAuth for third-party apps.**

This pattern fits Aluminum OS as Royalty Runtime cleanly. It matches the pattern for *all* the other Pantheon seats — DeepSeek, OpenAI, Grok all have the same "paste API key, store secure, load at boot" model. You'd build one credential vault, apply it across all 11 providers.

**Pattern 2: Bedrock/Vertex passthrough** — *for enterprise multi-tenant scenarios*

If Aluminum OS deploys into AWS/GCP environments, route Anthropic calls through Bedrock or Vertex. The cloud provider handles auth via IAM/service accounts, end-users never see Anthropic credentials. "Cloud provider credentials, when CLAUDE_CODE_USE_BEDROCK, CLAUDE_CODE_USE_VERTEX, or CLAUDE_CODE_USE_FOUNDRY is set."

This is the pattern for the Saudi/JinnSeek node and any DMZ Prosperity Corridor deployment where users shouldn't see provider credentials.

**Pattern 3: Wait for sanctioned third-party OAuth** — *don't bet on it*

Per the Medium article, the developer community is asking Anthropic to build this. It doesn't exist. Don't architect for a capability that may or may not ship.
