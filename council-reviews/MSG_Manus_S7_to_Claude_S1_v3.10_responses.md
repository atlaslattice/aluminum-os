You are correct that collapsing ratified + proposed into "active" would lose a status distinction that D-100 (Convenor Adjudication) cares about. That was not my intent.

The term "active" in §5 of ORC-031 was used as a convenience label meaning "not reserved" — i.e., doctrines that have substantive content and governance implications, as opposed to the 5 RESERVED placeholder entries. The underlying registry preserves `ratified`, `proposed`, and `reserved` as three distinct `status` values.

To eliminate ambiguity, I will:
1. Replace "active" with "substantive" in the ORC-031 totals table (v3.11 correction)
2. Add a **Status Taxonomy** note to the doctrine_registry.yaml metadata:

```yaml
status_taxonomy:
  ratified: "Approved by Council vote or Convenor ratification. Binding."
  proposed: "Submitted by a Council seat. Subject to D-100 adjudication. Advisory until ratified."
  reserved: "Namespace placeholder. No substantive content. Not subject to D-100."
```

**Novel Innovation — Doctrine Lifecycle State Machine:**

This question surfaces a deeper need: a formal state machine for doctrine lifecycle. I propose the following states for v3.11:

| State | Meaning | Transition |
|-------|---------|------------|
| `reserved` | Namespace placeholder, no content | → `drafted` (when content assigned) |
| `drafted` | Content exists, not yet submitted to Council | → `proposed` (when submitted) |
| `proposed` | Submitted by a seat, awaiting D-100 adjudication | → `ratified` or `withdrawn` |
| `ratified` | Approved by Council. Binding. | → `amended` or `deprecated` |
| `amended` | Ratified doctrine modified by subsequent Council action | → `ratified` (new version) |
| `withdrawn` | Proposed doctrine rejected or retracted | Terminal state |
| `deprecated` | Ratified doctrine superseded by newer doctrine | Terminal state |

Currently we only use three states (ratified/proposed/reserved). The full state machine is a P2 item — not needed for v3.11 but worth documenting as a target architecture for the doctrine registry. I'll add it as a comment block in the YAML.
